## Analysis

### Strengths

- **All basic commands beat Redis**: PING, SET, GET, INCR, LPUSH, RPUSH, LPOP, RPOP, HSET all show 104–113% of Redis throughput.
- **Outstanding LRANGE_100 performance**: 59,102 vs 43,048 req/s (137.3%) — avacado's list implementation is significantly faster for small range reads.
- **p99 latency is excellent**: All commands except LRANGE_600 have lower p99 than Redis, indicating more consistent performance under load.
- **INCR p99 (0.711ms) vs Redis (0.919ms)**: 22% lower tail latency — the single-goroutine executor eliminates lock contention effectively.

### Areas for Improvement

- **LRANGE at large sizes (≥300 elements)**: Avacado's throughput matches Redis at 300 elements but falls behind at 500 and 600 elements (89.5% and 83.3%). Latency degrades faster than Redis at large ranges.
  - Hypothesis: Per-element allocation or copy overhead in LRANGE response serialization; Redis may use more cache-friendly bulk copy.
- **All commands remain below 100K ops/s goal**: Peak is 84,317 (PING_INLINE). The single-goroutine executor serializes all operations — there's inherent throughput ceiling from this design.

### Bottleneck Hypotheses

1. **LRANGE scaling**: The listpack-based list implementation may allocate more GC pressure for large slices returned by LRANGE. Redis's internal list encoding likely reuses memory more efficiently.
2. **Single-goroutine executor ceiling**: All commands pass through one channel+goroutine, capping maximum throughput. For the current level, this isn't a bottleneck (we beat Redis), but at higher concurrency it could become one.

## Recommendations

### Optimization Priorities

1. **Investigate LRANGE large-range degradation**: Profile with `pprof` at LRANGE_500+ to identify whether the bottleneck is allocation, copying, or serialization. Target: match Redis at 500+ elements.
2. **LRANGE serialization**: Check whether the response for LRANGE allocates a fresh `[]byte` slice per element vs. bulk copy — a preallocated write buffer would reduce GC pressure.
