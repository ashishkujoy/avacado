## Analysis

### Strengths

1. **Throughput Exceeds Redis**
   - SET: 6.6% faster than Redis (88,889 vs 83,403 req/s)
   - GET: Essentially identical to Redis (99.4%)
   - Represents a major improvement from prior 62–70% ratios

2. **Superior Tail Latency**
   - SET p99: 34% lower than Redis (0.767ms vs 1.167ms)
   - GET max: 20% lower than Redis (0.967ms vs 1.215ms)
   - SET p95: 13.5% lower than Redis
   - Consistent delivery under percentile pressure

3. **Overall Latency Parity**
   - 8 out of 10 latency metrics are equal or better than Redis
   - The one exception (GET avg, +2.5%) is within noise

### Context

The absolute throughput numbers are lower than February 2026 runs (~120K req/s for avacado, ~190K req/s for Redis). Redis in this run measured ~87K req/s — roughly half its February baseline. This suggests the system was under elevated load during benchmarking. However, both servers ran under identical conditions, making the **ratio the valid comparison metric**.

## Performance vs Historical Baselines

| Date | Avacado SET | Redis SET | Ratio |
|------|------------:|----------:|------:|
| 2026-02-09 (initial) | 117,786 | 189,036 | 62.3% |
| 2026-02-20 (post-refactor) | 119,617 | 193,424 | 61.8% |
| 2026-02-20 (full suite) | 121,065 | 173,010 | 70.0% |
| **2026-04-23 (this run)** | **88,889** | **83,403** | **106.6%** |

The ratio improvement from 62–70% → 100%+ indicates a meaningful architectural change. Most likely contributors: HDEL implementation, parser refactor (e91d81b removes redundant parser code), and accumulated protocol handling improvements since February.

---

## Recommendations

1. **Re-run on an idle machine** to get absolute throughput numbers without system load interference.
2. **Concurrency scaling test**: now that parity is achieved at 50 clients, test 100, 500, 1000 clients to verify scaling holds.
3. **Extended command suite**: benchmark INCR, LPUSH, RPUSH, LPOP, RPOP, HDEL to verify parity extends across the command surface.
